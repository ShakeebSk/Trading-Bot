# Binance Futures Testnet Trading Bot

A small, structured Python CLI that places **MARKET** and **LIMIT** orders on
Binance **USDT-M Futures Testnet**, with input validation, structured
logging, and clean error handling.

Talks to the API with direct signed REST calls (`requests` + HMAC-SHA256) —
no `python-binance` dependency — so every request/response is fully
transparent and logged.

## Project Structure

```
trading_bot/
  bot/
    __init__.py
    client.py          # Binance Futures Testnet REST client (signing, requests, error handling)
    orders.py           # Order placement orchestration (request -> client -> normalized result)
    validators.py        # CLI input validation (symbol/side/type/quantity/price)
    logging_config.py     # Rotating file + console logging setup
  tests/
    test_validators.py    # Unit tests for validation logic (no network needed)
  logs/
    sample_bot.log         # Example log output (see "About the sample logs" below)
  cli.py                    # CLI entry point (argparse)
  requirements.txt
  .env.example
  README.md
```

## Setup

1. **Clone/download this repo** and `cd` into it.

2. **Create a virtual environment** (recommended):
   ```bash
   python3 -m venv venv
   source venv/bin/activate   # Windows: venv\Scripts\activate
   ```

3. **Install dependencies:**
   ```bash
   pip install -r requirements.txt
   ```

4. **Get Binance Futures Testnet API credentials:**
   - Register/log in at https://testnet.binancefuture.com
   - Generate an API key + secret from the testnet dashboard
   - (The testnet gives you fake USDT funds automatically for paper trading.)

5. **Configure credentials.** Copy `.env.example` to `.env` and fill it in:
   ```bash
   cp .env.example .env
   ```
   ```
   BINANCE_API_KEY=your_testnet_api_key_here
   BINANCE_API_SECRET=your_testnet_api_secret_here
   ```
   Alternatively, export them directly as shell env vars, or pass
   `--api-key` / `--api-secret` on the command line for a one-off run.

## How to Run

### Market order (BUY)
```bash
python cli.py --symbol BTCUSDT --side BUY --type MARKET --quantity 0.01
```

### Limit order (SELL)
```bash
python cli.py --symbol ETHUSDT --side SELL --type LIMIT --quantity 1 --price 3500.5
```

### Full option list
```bash
python cli.py --help
```

| Flag               | Required        | Description                                  |
|---------------------|-----------------|-----------------------------------------------|
| `--symbol`           | yes             | Trading pair, e.g. `BTCUSDT`                  |
| `--side`             | yes             | `BUY` or `SELL`                               |
| `--type`             | yes             | `MARKET` or `LIMIT`                           |
| `--quantity`          | yes             | Order quantity                                |
| `--price`             | for LIMIT only   | Limit price                                   |
| `--time-in-force`      | no (default GTC) | `GTC`, `IOC`, or `FOK`                        |
| `--api-key`           | no (or via env)  | Overrides `BINANCE_API_KEY`                   |
| `--api-secret`         | no (or via env)  | Overrides `BINANCE_API_SECRET`                |
| `--base-url`           | no               | Defaults to the Futures Testnet URL           |

### What you'll see

```
--- Order Request Summary ---
  Symbol:        BTCUSDT
  Side:          BUY
  Type:          MARKET
  Quantity:      0.01
------------------------------

--- Order Response ---
  Order ID:      3450001234
  Status:        FILLED
  Executed Qty:  0.010
  Avg Price:     67540.20
-----------------------

 SUCCESS: order placed.
```

On failure (bad input, API rejection, or network issue) you get a clear
` FAILED` message with the reason, and a non-zero exit code
(`1` = validation error, `2` = API/network error while placing the order).

## Logging

All requests, responses, and errors are logged to `logs/bot.log`
(rotating, 2 MB per file, 5 backups kept). Console output stays at INFO+ so
the terminal isn't noisy, while the file captures full DEBUG-level detail —
including the exact request parameters sent and the raw response body.

API signatures are **redacted** in all logs (both in the structured request
params and in raw exception text), so log files are safe to share/submit.

### About the sample logs

`logs/sample_bot.log` was generated by exercising the real client/orders/
logging code path with the HTTP layer stubbed to return realistic Binance
response payloads (see `scripts` in git history / can be regenerated with a
small script that patches `requests.Session.request`). This was necessary
because **this development environment's network egress does not include
`testnet.binancefuture.com`** — the same code, run from a normal machine
with real testnet credentials, produces the same log format against the
live API. Every line was written by the actual `logging_config.py` /
`client.py` / `orders.py` modules, not hand-written — only the HTTP response
was simulated. Run the CLI yourself with real testnet credentials to
generate a first-party `logs/bot.log`; two runs (one MARKET, one LIMIT) are
enough to satisfy the deliverable.

## Error Handling & Validation

- **Input validation** (`bot/validators.py`): symbol format, side/type
  whitelist, positive numeric quantity, price required for LIMIT — all
  checked *before* any network call, with clear messages.
- **API errors** (`BinanceAPIError`): non-2xx responses are caught, logged
  with status code + Binance error code/message, and surfaced as a clean
  failure message.
- **Network errors** (`BinanceNetworkError`): timeouts, DNS failures,
  connection errors are caught separately so the user knows it's a
  connectivity issue, not a rejected order.
- **Unexpected errors**: a final catch-all in `bot/orders.py` logs the full
  traceback (`logger.exception`) but still returns a controlled failure
  result — the CLI never crashes with a raw stack trace.

## Testing


## Assumptions

- Only **USDT-M Futures Testnet** is targeted (`https://testnet.binancefuture.com`),
  as specified in the task. Coin-M futures and Spot testnet are out of scope.
- Quantity/price precision (lot size, tick size) is **not** pre-validated
  against exchange filters client-side; Binance's own filter validation on
  the server is relied on, and any rejection is surfaced as a normal API
  error. A production version would fetch `/fapi/v1/exchangeInfo` and
  round to the correct step size before sending.
- `GTC` is the default time-in-force for LIMIT orders since it wasn't
  specified as a required CLI input; it's overridable via `--time-in-force`.
- Credentials are read from `.env`/environment variables by default (never
  hardcoded), with CLI flags as an override for convenience/testing.
- No bonus feature was implemented (kept the core scope tight and fully
  tested/working) — happy to add Stop-Limit, an enhanced interactive CLI
  (prompt_toolkit/Typer), or a lightweight UI as a follow-up if useful.

## Security Note

`.env` is git-ignored. Never commit real API keys. Testnet keys have no
real-money value, but the same hygiene applies as production keys.
